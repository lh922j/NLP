# 한국어 토크나이저 비교 평가 리포트

---

## 1. 개요

본 리포트는 한국어 뉴스 도메인 데이터를 활용하여 BPE, WordPiece 토크나이저를 직접 학습하고, 한국어 형태소 분석기(Mecab, Okt, Kkma)와 함께 성능을 비교 평가한 결과를 정리한 것입니다.

---

## 2. 데이터셋

### 2.1 학습 데이터

| 항목 | 내용 |
|------|------|
| 출처 | 국방일보 (2024.05 ~ 2025.04, 월별 PDF 12개) |
| 수집 방법 | PyMuPDF(fitz)로 PDF 단어 단위 추출 후 줄 재구성 |
| 전체 용량 | 약 29.7 MB |
| 데이터 형태 | 줄 단위 텍스트 (단어 경계 공백 복원 완료) |
| 도메인 | 군사·안보·국방 관련 한국어 신문 기사 |

### 2.2 전처리

- `page.get_text("words")`로 단어 단위 추출 → y 좌표 기준 같은 줄 묶기 → 공백으로 연결
- 기존 `page.get_text("text")` 방식은 어절 간 공백이 소실되는 문제가 있어 교체
- 전체 데이터를 줄 단위로 분리 후 랜덤 셔플 (seed=42)

## 3. 실험 설정

### 3.1 BPE 설정

| 항목 | 값 |
|------|----|
| 목표 어휘 크기 (vocab_size) | **7,000** |
| 최소 빈도 (min_frequency) | 1 |
| Pre-tokenizer | Whitespace (공백 기준 분리) |
| 특수 토큰 | `[UNK]`, `[PAD]`, `[CLS]`, `[SEP]`, `[MASK]` |
| 라이브러리 | HuggingFace `tokenizers` |

> **BPE vocab_size를 7,000으로 설정한 이유**: BPE는 학습 시 코퍼스의 모든 고유 음절을 초기 어휘로 등록한 뒤 병합(merge)을 수행합니다. 학습 데이터 100% 기준 고유 음절 수가 약 6,570개로 확인되었으며, vocab_size가 이 값 이하이면 merge 공간이 없어 글자 단위 토크나이저로 퇴화합니다. 이를 방지하기 위해 merge가 가능한 최솟값인 7,000으로 설정하였습니다.

### 3.2 WordPiece 설정

| 항목 | 값 |
|------|----|
| 목표 어휘 크기 (vocab_size) | **12,000** |
| 최소 빈도 (min_frequency) | 2 |
| Pre-tokenizer | Whitespace (공백 기준 분리) |
| 특수 토큰 | `[UNK]`, `[PAD]`, `[CLS]`, `[SEP]`, `[MASK]` |
| 라이브러리 | HuggingFace `tokenizers` |

> **WordPiece vocab_size를 12,000으로 설정한 이유**: WordPiece는 단어 내부 연속 서브워드에 `##` 접두사를 붙여 독립 토큰으로 저장합니다. 예를 들어 `국`과 `##국`은 별개 토큰입니다. 따라서 동일 코퍼스에서도 BPE 대비 약 2배의 초기 어휘가 필요합니다. 실험에서 vocab_size=10,000으로 설정했을 때 100% 학습 데이터에서 vocab이 ~11,700으로 초과되어 merge가 발생하지 않았고, 이를 기반으로 merge가 가능한 최솟값인 12,000으로 설정하였습니다.

### 3.3 형태소 분석기

| 분석기 | 특징 |
|--------|------|
| Mecab | 사전 기반 형태소 분석, 속도 빠름 |
| Okt | KoNLPy 기반, 규칙+통계 혼합 방식 |
| Kkma | KoNLPy 기반, 문법 규칙 기반 정밀 분석 |

> 형태소 분석기는 별도 학습 없이 평가셋에 대한 성능만 비교합니다.

### 3.4 평가 지표

| 지표 | 설명 |
|------|------|
| 어휘 크기 (Vocab Size) | 학습된 고유 토큰 수 |
| OOV 비율 (%) | 평가셋에서 `[UNK]`로 처리된 토큰 비율 |
| Fertility | 토큰 수 ÷ 공백 기준 어절 수 (높을수록 잘게 분리) |
| 속도 (k tokens/sec) | 초당 토크나이징된 토큰 수 |

## 4. 실험 결과

### 4.1 학습 데이터 크기별 어휘 크기 변화

| 학습 비율 | BPE vocab | WordPiece vocab |
|-----------|-----------|-----------------|
| 10% | 7,000 | 12,000 |
| 50% | 7,000 | 12,000 |
| 100% | 7,000 | 12,000 |

**분석**

- BPE와 WordPiece 모두 모든 학습 비율에서 목표 vocab_size에 도달합니다.
- 데이터 크기에 따른 어휘 크기 변화가 없는 이유는, 이미 10% 학습 데이터에서도 고유 음절 수(~3,470개)가 충분하여 merge를 통해 vocab_size를 채울 수 있기 때문입니다.
- 이는 한국어 음절 문자 체계의 높은 다양성에서 기인합니다. 학습 데이터가 증가할수록 새로운 희귀 음절이 추가되지만, 그보다 많은 merge 기회가 생겨 결과적으로 항상 목표 vocab에 수렴합니다.

---

### 4.2 학습 데이터 크기별 OOV 비율 변화

| 학습 비율 | BPE OOV (%) | WordPiece OOV (%) |
|-----------|-------------|-------------------|
| 10% | 0.00 | 0.00 |
| 50% | 0.00 | 0.00 |
| 100% | 0.00 | 0.00 |

**분석**

- 두 토크나이저 모두 모든 조건에서 OOV 0%입니다.
- 서브워드 알고리즘은 초기 어휘에 코퍼스의 모든 고유 글자를 포함하므로, 동일 도메인 평가셋에서는 미등록 글자가 존재하지 않습니다.
- 평가셋이 학습 코퍼스와 동일한 국방일보 도메인에서 추출되었기 때문에 도메인 외 어휘가 없어 OOV 0%가 유지됩니다.

### 4.3 전체 학습 데이터 기준 Fertility 비교

| 토크나이저 | Fertility |
|-----------|-----------|
| BPE (100%) | 10.235 |
| WordPiece (100%) | 10.639 |
| Mecab | 8.212 |
| Okt | **6.991** |
| Kkma | 9.180 |

**학습 데이터 크기별 BPE/WP Fertility 변화**

| 학습 비율 | BPE Fertility | WP Fertility |
|-----------|--------------|--------------|
| 10% | 7.966 | 7.739 |
| 50% | 8.866 | 8.765 |
| 100% | 10.235 | 10.639 |

**분석**

- 형태소 분석기(Okt: 6.991, Mecab: 8.212)가 서브워드 토크나이저보다 낮은 fertility를 보입니다. 언어 규칙 기반으로 의미 단위를 더 잘 보존하기 때문입니다.
- **데이터가 많을수록 fertility가 높아지는 역설적 현상**이 관찰됩니다. 이는 학습 데이터가 증가할수록 새로운 희귀 음절이 추가되어 고유 음절 수가 증가하고, 그로 인해 merge에 할당되는 vocab 공간이 줄어들기 때문입니다.
  - 10%: 고유 음절 ~3,470개 → merge 여유 3,530개 → 압축 효율 높음 (fertility 7.97)
  - 100%: 고유 음절 ~6,570개 → merge 여유 430개 → 압축 효율 낮음 (fertility 10.24)
- WordPiece가 BPE보다 fertility가 소폭 높은 이유는 `##` prefix 처리 방식 때문입니다.
- BPE(10%) fertility 7.97은 Mecab(8.21)보다 낮아, 충분한 merge가 이루어질 때 서브워드 토크나이저도 형태소 분석기 수준의 압축 효율을 달성할 수 있음을 보여줍니다.

---

### 4.4 전체 학습 데이터 기준 속도 비교

| 토크나이저 | 속도 (k tokens/sec) | 상대 속도 (vs Kkma) |
|-----------|---------------------|---------------------|
| BPE (100%) | 811.9 | 약 2,030배 |
| WordPiece (100%) | 388.7 | 약 972배 |
| Mecab | 4.7 | 약 12배 |
| Okt | 0.7 | 약 1.8배 |
| Kkma | 0.4 | 기준 |

**분석**

- BPE(811.9k tok/s)가 압도적으로 빠릅니다. HuggingFace `tokenizers` 라이브러리는 Rust 기반으로 구현되어 있어 Python 오버헤드가 거의 없습니다.
- WordPiece(388.7k)는 BPE보다 느린데, `##` prefix 처리 등 내부 로직이 BPE보다 복잡하기 때문입니다.
- Mecab(4.7k)은 형태소 분석기 중 가장 빠릅니다. 사전 기반 탐색 방식으로 Java JVM 없이 동작하기 때문에 Okt, Kkma 대비 월등한 속도를 보입니다.
- Okt(0.7k)와 Kkma(0.4k)는 Java 기반 KoNLPy를 사용하므로 JVM 오버헤드가 발생합니다. 특히 Kkma는 문법 규칙 기반의 정밀 분석을 수행하기 때문에 가장 느립니다.

## 6. 토크나이징 예시

동일한 문장을 각 토크나이저로 분석한 결과입니다.

**입력 문장**: `"대한민국 육군은 최전방에서 임무를 수행한다"`

| 토크나이저 | 토크나이징 결과 | 토큰 수 |
|-----------|----------------|--------|
| BPE | `대한`, `민국`, `육군`, `은`, `최전`, `방에서`, `임무`, `를`, `수행`, `한다` | 10 |
| WordPiece | `대한`, `##민국`, `육군`, `##은`, `최전`, `##방`, `##에서`, `임무`, `##를`, `수행`, `##한다` | 11 |
| Mecab | `대한민국`, `육군`, `은`, `최전방`, `에서`, `임무`, `를`, `수행`, `한다` | 9 |
| Okt | `대한민국`, `육군`, `은`, `최전방`, `에서`, `임무`, `를`, `수행`, `한다` | 9 |
| Kkma | `대한민국`, `육군`, `은`, `최전방`, `에서`, `임무`, `를`, `수행`, `하`, `ㄴ다` | 10 |

> BPE/WordPiece는 `대한민국`을 `대한` + `민국`으로 분리하는 반면, 형태소 분석기는 고유명사로 인식해 하나의 토큰으로 처리합니다.
> WordPiece는 단어 내부 서브워드에 `##` prefix를 붙여 경계를 표시합니다.
> Kkma는 `한다`를 `하` + `ㄴ다`로 분리하는 등 가장 세밀한 형태소 분석을 수행합니다.

## 7. 결론

1. vocab_size 설정의 중요성 : BPE는 고유 음절 수 이하의 vocab_size에서 merge가 전혀 발생하지 않아 글자 단위 토크나이저로 퇴화합니다. WordPiece는 `##` 접두사 토큰으로 인해 동일 코퍼스에서도 약 2배의 vocab_size가 필요합니다. 이는 한국어 음절 문자 체계의 특성에서 기인하며, vocab_size 설정 시 코퍼스의 고유 음절 수를 사전에 확인해야 합니다.

2. OOV 처리: BPE와 WordPiece 모두 서브워드 알고리즘 특성상 OOV를 완전히 제거합니다. 반면 형태소 분석기는 미등록어에 취약하며, 특히 군사·국방 도메인의 특수 용어(작전명, 무기체계명 등)에서 미등록어 문제가 발생할 수 있습니다.

3. Fertility와 학습 데이터 크기의 역관계: 본 실험에서 학습 데이터가 많을수록 BPE/WP fertility가 오히려 높아지는 현상이 관찰되었습니다. 이는 고정된 vocab_size 내에서 데이터가 많아질수록 고유 음절 수가 증가해 merge 가능 공간이 줄어들기 때문입니다. 충분히 큰 vocab_size를 확보한다면 이 현상은 해소됩니다.

4. 속도: BPE(811.9k tok/s)가 전체 토크나이저 중 가장 빠르며, 형태소 분석기 중에서는 Mecab(4.7k tok/s)이 JVM 없이 동작해 가장 빠른 속도를 보였습니다. Okt(0.7k)와 Kkma(0.4k)는 JVM 오버헤드로 실시간 서비스에 적합하지 않습니다.

5. 도메인 적합성: 국방 도메인 특수 용어 처리와 OOV 대응 측면에서는 BPE/WordPiece가 유리하고, 언어적 의미 단위 보존(낮은 fertility)과 추가 학습 불필요 측면에서는 Mecab/Okt가 유리합니다.
